1. From ML to Multimodal ML

The lecture re-grounds machine learning in Tom Mitchell's classic definition before extending it to multiple modalities.

Definition · Mitchell Exam

"A computer program is said to learn from experience E with respect to some class of tasks T and performance measure P, if its performance at tasks in T, as measured by P, improves with experience E."

Be ready to identify E, T, and P in a robot scenario — the slide poses this as an in-class exercise.

Worked example — iCub object recognition

SymbolWhat it isiCub instance
TTaskObject recognition with color and depth data
EExperienceThe iCub multisensor dataset
PPerformance measureAccuracy, precision, recall, F1 score
Definition · Multimodal ML Exam

"Multimodal Machine Learning is the study of computer algorithms that learn and improve through the use and experience of data from multiple modalities."

Source: CMU MMML course (Fall 2023).

2. What Is a Modality?
Definition · core Exam

A modality is "sensory data that are associated with different aspects of the observed phenomena."

For humans, the canonical list is the five senses — sight, hearing, touch, taste, smell — plus proprioception (the sense of where your body parts are in space). For a robot the modalities map to its sensor suite: color cameras, depth sensors, microphones, tactile pads, joint encoders.

Why we need multimodal learning Exam

Two reasons recur on the slides — memorise both:

  1. To form a robust sensory representation — if one modality is noisy or missing, others compensate.
  2. To leverage the complementary characteristics of modalities — depth disambiguates what color cannot, audio confirms what touch suggests.
Concept · brain analogy Prof

The slide illustrates this with a perception pipeline: object → sensory organs (color, depth, tactile, taste) → cortical processing → unified representation. The cortex is the integrator. A multimodal robot is the engineering analogue.

3. Five Challenges in Multimodal ML

From the Baltrušaitis, Ahuja & Morency (2018) survey — a canonical taxonomy and a likely exam target.

Taxonomy · five challenges Exam
  1. Representation — how to represent multimodal data (pixels, signals, symbols, etc.). ★ focus of this lecture.
  2. Translation — how to map data from one modality to another (e.g. caption an image, describe a sound).
  3. Alignment — how to identify direct relations between modalities (which word lines up with which video frame).
  4. Fusion — how to join information from two or more modalities (data level, decision level, intermediate). ★ focus of this lecture.
  5. Co-learning — how to transfer knowledge between modalities (a label-rich modality teaching a label-poor one).

The slide explicitly says: "We will focus on representation and fusion." The other three are named so you recognise them on the exam — you do not need to reproduce algorithms for them.

The slide also notes the deck uses "representation" and "features" interchangeably — a small but exam-relevant terminology choice. Source: Baltrušaitis, T., Ahuja, C., & Morency, L. P. (2018). Multimodal machine learning: A survey and taxonomy. IEEE TPAMI 41(2), 423–443.

4. Representation — From Raw Data to Vectors

Before fusion you need a representation of each modality. The slides emphasise that what your robot "sees" is not what you see: a color image is a raw matrix of RGB values, a colorized depth map is another matrix, audio is a 1-D waveform, text is a token sequence.

The slides ask, repeatedly:

  • How can we represent text data as a vector?
  • How can we represent speech as a vector?
  • How can we represent color and depth as a vector?

This is the representation challenge from the taxonomy — and once everything is a vector, you can fuse.

Hand-crafted vs learned representations Exam

ApproachHow features are madeExamples
Hand-crafted Designed by humans using classical algorithms SIFT, SURF, HOG for images; OpenCV's feature detectors
Learned (non-hand-engineered) Extracted automatically by a deep network CNN features for images; AENet for audio
Desired specifications Prof

A good multimodal representation must:

  1. Map similar concepts to similar points — different carrots cluster together, far from cars, in the embedding space.
  2. Be robust to noise — small input perturbations should not flip the prediction.
  3. Have a strategy for missing data — when one modality drops out (sensor fault, occlusion) the system must still respond.

The slide refers to CNN-Explainer for an interactive view of how convolutional layers build features layer-by-layer — useful intuition for why "automatic feature learning" gradually replaced hand-crafted SIFT/HOG.

5. Toy Example — iCub Multimodal Object Recognition

The whole second half of the lecture is built around one running example: the iCub humanoid recognising household objects from five sensor streams.

iCub sensor suite

  • Two color cameras in the eyes (left + right).
  • One Realsense D435i mounted just above the eyes — provides RGB + depth.
  • Touch sensor in the left hand.
  • Microphone.

This gives five modalities: iCub left camera, iCub touch, Realsense color, Realsense depth, and iCub microphone.

Unimodal baseline

The naïve approach — train five separate models, one per modality. Each outputs a discrete probability distribution over objects. The slides show this five-model layout repeating across slides 27–32 to drill the picture.

Diagnosis · why unimodal breaks Prof

"Two of the five models cannot correctly recognise the objects. Why?"

External reasons:

  • Noise in the environment.
  • Miscalibrated sensors.

Model-related reasons:

  • Wrong model selection.
  • Non-regularised weights.
  • Using raw data as input (no feature extraction).

This motivates both reasons for multimodal integration introduced earlier — robustness and complementarity. Multiple weak streams > one strong stream when conditions vary.

6. Multimodal Fusion — Three Strategies

This is the core technical content of the lecture and the most exam-likely block. Multimodal fusion is the fourth challenge in the Baltrušaitis taxonomy — joining information from two or more modalities. There are three places in the pipeline where you can do it.

Three fusion strategies Exam
StrategyWhere the fusion happensAlso called
a) Data-levelBefore the ML algorithm — fuse the inputsEarly fusion
b) Decision-levelAfter the ML algorithm — fuse the model outputsLate fusion
c) IntermediateInside the model — fuse intermediate layers / representations(deep-only)

Source: Deep Multimodal Learning: A Survey on Recent Advances and Trends — referenced repeatedly in the slides. Note that intermediate fusion only exists for deep models (it requires intermediate-layer activations to fuse).

7. Data-Level (Early) Fusion

The simplest strategy. Pre-process each modality, glue the inputs together, feed one model.

Mechanics — what does "glue together" mean?

  • Concatenate raw inputs — stack the vectors / matrices.
  • Reduce input dimension — PCA or similar before concatenation.
  • Use hand-crafted features from each modality, then concatenate.

You then train one model on the fused input and read off a single discrete probability distribution over objects.

Hidden assumption Exam

Data-level fusion implicitly assumes conditional independence among modalities. The slide flags this with a question: "depth and color (?)" — are color and depth really conditionally independent? They are not — they describe the same scene from the same viewpoint. Data-level fusion gets away with this in practice but it is a known weakness.

Other limitations: all modalities must share compatible dimensions after preprocessing, and you commit to one model architecture for the whole stack — so you cannot use a CNN for images and an SVM for depth at the same time.

8. Decision-Level (Late) Fusion

The opposite of data-level fusion — train one model per modality, let each emit its own decision, then combine the decisions.

Mechanics

  • Employ different (or the same) models for different modalities.
  • Collect decisions from each separately-trained model.
  • Fuse the decisions by averaging, sum-then-max, or meta-learning.
  • Recognise the family — this is ensemble learning.
When to use decision-level fusion Exam
  • The modalities are uncorrelated.
  • The modalities have different dimensions (you can't easily concatenate).
  • You want to exploit different ML models per modality — e.g. CNNs for image, SVM for depth, MLPs for touch.

The toy diagram on slide 41 shows it plainly: five sensors → five models (Model 1…5) → five decisions → one fused final output. Each branch is independent up to the last step.

Trade-off: decision-level fusion throws away cross-modal correlations that might exist before the decision step. If color and depth share information, late fusion misses it.

9. Intermediate Fusion

The deep-learning sweet spot — fuse inside the model rather than at the inputs or outputs.

Mechanics

  • Each modality enters its own subnetwork (CNN for color, MLP for depth, etc.).
  • At some intermediate layer, the per-modality activations are projected into a shared / joint representation.
  • A final dense layer maps the joint representation to outputs.
  • Features are non-hand-crafted — learned end-to-end from data.
Design heuristic Prof

Fuse similar modalities together. The slide groups e.g. (left camera + Realsense color) before fusing with (depth + touch). The architecture decision is "which modalities are siblings?" — that determines where in the network they meet.

The slide references a Keras visualisation with CNNs for color and MLPs for depth as a concrete example of an intermediate-fusion architecture.

Why intermediate often wins

You get the best of both worlds: per-modality feature learning (like decision-level) plus cross-modal interaction in the joint representation (like data-level). The shared representation can absorb correlations that late fusion ignores.

10. Use Case — Rotation-Invariant Object Recognition

The lecture closes on a research result from Kirtay & Hafner: comparing decision-level vs intermediate fusion for rotation-invariant object recognition.

Three open questions surveyed by the literature (slide 44):

  1. Which is the better fusion method?
  2. Which modalities should be fused together?
  3. How can we determine the model architecture?

The toy experiment fed a random-noise camera input to test how each fusion method holds up when one modality fails.

Result · two implications Exam

The slides explicitly tie the result back to the two motivations introduced at the start:

  1. Robust sensory representation. Random noise in the camera did not destroy the intermediate-fusion architecture's accuracy — the other modalities carried it.
  2. Complementary characteristics. Unimodal models already achieve high accuracy. Combining them into a multimodal architecture increases accuracy further.

In the head-to-head, intermediate fusion outperformed decision-level fusion on this task — the joint representation was more robust to single-modality corruption than averaging independent decisions.

The interaction & communication framing

The lecture's title also names Interaction and Communication. The slide deck threads this lightly via the "Hey iCub! Bring me the object" motif — the robot needs to fuse language (spoken request) with vision and depth (find the object) and proprioception/touch (grasp it). Multimodal fusion is the substrate for grounded human-robot communication. The Week 08 lecture (Affective & Social HRI / RRI) builds directly on this.

Key Takeaways
  1. Mitchell's T/E/P still defines learning. For a multimodal robot: T = task (object recognition), E = the multisensor dataset, P = accuracy / precision / recall / F1.
  2. Modality = sensory data about one aspect of a phenomenon. For robots: color, depth, touch, audio, proprioception.
  3. Two reasons for multimodal learning — robust representation + complementary modalities. Memorise both.
  4. Five MMML challenges (Baltrušaitis): representation, translation, alignment, fusion, co-learning. The lecture focuses on representation and fusion.
  5. Hand-crafted (SIFT/SURF/HOG) vs learned (CNN/AENet) features — modern multimodal pipelines favour the latter.
  6. Three fusion strategies: data-level (early, fuse inputs), decision-level (late, fuse outputs / ensemble), intermediate (fuse inside the network in a joint representation).
  7. Decision-level fusion fits when modalities are uncorrelated, have different dimensions, or warrant different models per modality (CNN for image, SVM for depth, MLP for touch).
  8. Intermediate fusion empirically wins the rotation-invariant object recognition use case — and stays robust when one modality is replaced by random noise.
