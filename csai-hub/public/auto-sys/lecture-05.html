1. What Is Adaptivity?


      
The lecture opens with a (famously misattributed) Darwin quote: "It is not the strongest of the species that survives, nor the most intelligent, but the one most adaptable to change." The prof points out the quote is an internet legend — Darwin never wrote it. The sentiment, however, frames the week: adaptivity is what we want to engineer into autonomous systems.


      

        
Definition · core Exam

        
Adaptivity is "to develop the behavioral and cognitive skills required to perform a desired function." (Nolfi, Behavioral and Cognitive Robotics)

        
"More adaptive = more intelligent" — for the purposes of this course.

      


      
Methods to achieve adaptivity

      
    
        
  • Evolutionary methods — population-level adaptation
    • 
        
  • Learning from Demonstration (LfD) — copy a human teacher
    • 
        
  • Reinforcement Learning (RL) — trial-and-error, reward-guided
    • 
        
  • … and others (developmental, intrinsic motivation, etc.)
    • 
      


      
This lecture covers LfD and RL. The exam frequently mixes adaptivity terminology with the operant conditioning callback from Week 1 (Skinner) — recognise reward-based learning as the bridge.


      

        
Prof emphasis Prof

        
Two competing schools of thought: "Reward is enough" (Silver / DeepMind) — reward suffices for intelligence. "Reward is not enough" — other mechanisms (curiosity, intrinsic drives) are required. The prof refuses to commit to either camp; expect this distinction on the exam.

      

    

  


  

    

      
      
      
    


    

      
2. Learning from Demonstration (LfD)


      
Learning from Demonstration (a.k.a. imitation learning, programming by demonstration) is a paradigm for enabling robots to autonomously perform new tasks without analytical decomposition or hand-coded behavior. The robot's controller is derived from observations of a human's own performance.


      
The general workflow is always the same:


      

        
General LfD pipeline Exam

        
Demonstrate → Train a model → Evaluate

        
You need a robot capable of the task and a demonstrator (operator) who can effortlessly perform it.

      


      
Why we need LfD

      
    
        
  • Formal description of tasks is hard — recall inverse kinematics from earlier weeks: it can have one, many, or no solutions, and is painful to derive analytically.
    • 
        
  • Daily activities are easy for humans but computationally expensive for robots (folding shirts, flipping omelets).
    • 
        
  • Not enough roboticists exist to write controllers for every possible task. LfD lets end-users teach robots — expert knowledge is not necessary.
    • 
        
  • Robots become flexible / re-trainable — same robot, different demonstrations, different tasks.
    • 
      


      

        
Prof's economic argument Prof

        
"What's the salary of a senior robotic software engineer?" Hand-coding a kitchen robot to open a washing machine, grasp varied dishware, and place items would take a PhD student "forever". Roboticists are expensive — LfD amortizes their effort across tasks.

      

    

  


  

    

      
      
      
    


    

      
3. Three Types of LfD


      
The prof was emphatic: "In the exam I will not accept 'kinematic teaching'. It is kinesthetic." Spelling matters. Prof


      

        
TypeHow it worksExample

        
          

            Kinesthetic teaching
            Robot's actuators in passive mode; experimenter physically manipulates the robot to force the desired behavior.
            Grab a robot arm by hand, guide it through a pouring motion, record joint trajectories.
          

          

            Teleoperation
            Actuators controlled by experimenter through a joystick / haptic device / web slider / VR rig. No direct contact with the robot.

4. The RL Framework


      
Reinforcement Learning is reward-guided trial-and-error learning in which an agent learns complex behaviors (e.g. navigation) by interacting with the environment. The prof's framing of when to use RL is exam-critical:


      

        
RL is for problems that are… Exam

        
Easy to evaluate, hard to solve.

        
Examples: a soccer match (look at the scoreboard — easy to evaluate; playing well — hard). Chess (checkmate is binary; playing well is combinatorial). Atari (the score is given; how to win is not).

      


      
The five core terms Exam

      

        
TermSymbolMeaning

        
          

            Agent
            —
            Decision-maker. Robot, chatbot, gambling agent, stock-tracker, medical agent.
          

          

            Environment
            —
            Everything the agent interacts with. Navigation area, T-shape maze, board game.
          

          

            State
            s
            Observation the agent receives — sensory information. E.g. odometry + camera + sonar reading.
          

          

            Action
            a
            What the agent does. Constrained by hardware. "Move left 1 m", "ask card / open card".
          

          

            Reward
            r
            A numeric value that guides agent behavior. Can be −1, 0, 1 or any number — negative rewards are allowed.
          

        
      


      

        
Exam trap · sign of reward Exam

        
True/false: "Reward must be positive." → FALSE. Students wrongly associate reward with their grades. Reward can be negative, zero, or positive. A typical scheme: +5 for goal, −2 for hitting an obstacle, 0 for an idle step.

      


      
State / action design choices

      
    
        
  • State and action definitions depend on your approach.
    • 
        
  • Spaces can be continuous (real-valued joint angles, real positions) or discrete (grid-based; "left / right / up / down").
    • 
        
  • The environment can be dynamic: moving obstacles, objects the robot itself displaces, wind / noise.
    • 
      


      
Goal of the agent: learn the actions that maximize the (cumulative discounted) reward.

    

  


  

    

      
      
      
    


    

      
5. The Markov Decision Process (MDP)


      
Every well-formed RL problem is described by an MDP. This is the single most exam-rich definition of the lecture.


      

        
MDP definition Exam

        
A Markov Decision Process is a tuple ⟨S, A, P, R, γ⟩:

        
    
          
  • S — set of possible states
    • 
          
  • A — set of possible actions
    • 
          
  • P — policy / state-action transition probability
    • 
          
  • R — reward function for each (state, action) pair
    • 
          
  • γ (gamma) — discount factor determining whether current or future reward is more valuable
    • 
        

      


      
Properties to remember

      
    
        
  • The number of states and actions can be finite or infinite.
    • 
        
  • The agent's behavior is defined by a policy π — a distribution of actions over the set of states.
    • 
        
  • The current state contains all information needed to choose the next action — history is irrelevant.
    • 
      


      

        
Markov (memoryless) property Exam

        
"The current state of the agent contains all necessary information about the world. The history of the state is not important to decide the next state." — "Make sure that you understand this property" (slide 33). Prof

        
Example (D. Silver, rephrased): in chess, you don't need to remember every previous move — the current state of the board is enough to decide the next move. Therefore chess satisfies the Markov property.

      


      
Fully vs partially observable

      

        
TypeDescriptionExample

        
          

            Fully observable MDP
            Current state contains all information.
            Chess (you see the whole board); a football near your foot with the goal visible.
          

          

            Partially observable MDP (POMDP)
            Some information is hidden — agent must infer state.
            Poker (opponent's cards are hidden); robot in fog with limited camera FOV.
          

        
      


      
The course holds you responsible for recognizing whether a given problem is Markov / fully observable, not for solving POMDPs. Prof

    

  


  

    

      
      
    


    

      
6. Policy & Value Function


      
Two interlocking concepts. Get them mixed up on the exam at your peril.


      

        
Definitions Exam

        
Policy π(s,a): a distribution over actions for each state — i.e. "in state s, the probability of choosing action a". The agent's behavior is the policy.

        
Optimal policy π*(s,a): a policy that maximises the expected cumulative reward. There may be more than one optimal policy.

        
Value function V(s) / Q(s,a): "how good a state — or a (state, action) pair — is under the policy π". It tells you how valuable being in s (and taking a) is in expectation.

      


      
Why both?

      
    
        
  • The policy says what to do.
    • 
        
  • The value function says how good doing it is.
    • 
        
  • Optimal policy ↔ optimal value function — they are tied: π* picks the action with the highest Q-value in every state.
    • 
      


      
In a 5×5 grid world, the goal is "find the optimal policy that reaches the red squares (food, charging) in the minimum number of actions." If the agent is dropped in the middle and the four red squares are at equal Euclidean distances, there are four optimal policies, each one valid.


      

        
Multiple optima Prof

        
"Optimal policy can be one, two, and multiple. Due to the noise in the environment, the robot will choose one of them and it will be a near-optimal solution." Real robots almost never sit on the true optimum — they orbit it.

      

    

  


  

    

      
      
    


    

      
7. Q-Learning & The Update Rule


      
Q-learning is the canonical value-based RL algorithm covered in this course. It maintains a table — one entry per (state, action) pair — and updates each entry based on observed rewards.


      

        
The Q-learning update rule Exam

        
Memorise this. The exam may ask you to identify the components, predict behavior when α or γ go to extremes, or apply it on a tiny grid.

        

          Q(s,a) ← Q(s,a) + α · [ r + γ · maxa' Q(s',a') − Q(s,a) ]
        

        
Where:

        
    
          
  • Q(s,a) — current value estimate of taking action a in state s
    • 
          
  • α (alpha) — learning rate — how much new information overrides old
    • 
          
  • r — reward received after taking action a
    • 
          
  • γ (gamma) — discount factor — importance of future reward
    • 
          
  • maxa' Q(s',a') — greedy estimate of best future value from the next state
    • 
          
  • The bracketed term is the TD (temporal-difference) error.
    • 
        

      


      
α and γ — the multiplicative trap Exam

      

        
ParameterValue = 0Value = 1Interpretation

        
          

            α (learning rate)
            Q never updates — agent learns nothing.
            Q is replaced entirely by the new estimate — old info discarded.
            Controls how aggressively the agent overwrites past experience.
          

          

            γ (discount factor)
            Agent fully myopic — only cares about immediate reward.
            Agent values future reward as much as immediate (no discount).
            Controls how far ahead the agent plans.
          

        
      


      

        
Prof's hint · slide 39 Prof

        
"Remember: it is multiplicative. What happens if it is 0 or 1?" The prof said this twice, once for α and once for γ. Treat it as a fill-in-the-blank.

      


      
Picking the action

      
Once the Q-table is trained, action selection is greedy: at every state, choose the action with the maximum Q-value:

      

        π*(s) = argmaxa Q(s,a)
      

      
In a small grid world, the values increase as the agent gets closer to the goal. This was the heat-map demo from the live grid demo — the gradient pushes the policy uphill toward reward.

    

  


  

    

      
      
      
    


    

      
8. Where Does Reward Come From? & The 5 Challenges


      
The provocative question: "Where does reward come from?" In humans, reward is generated internally by our biological system. In robots, reward is a signal from the environment — typically application-specific and hand-crafted by a domain expert.


      

        
Hand-crafted reward — the limit Prof

        
If your reward function is hand-crafted, your robot's intelligence is capped by your domain knowledge. The instant the environment exits your expertise (dynamic obstacles, novel objects, weather), the reward function fails — and so does the agent. Automatic / intrinsic reward generation is an open research problem (the prof's PhD topic).

      


      
Five RL challenges Exam

      

        
ChallengeWhat goes wrong

        
          

            Credit assignment problem
            Which past action led to the reward you just got? Same as backprop's "which weight contributed to this loss?" — but harder, because the "weights" are entire action histories.
          

          

            High number of states
            Real-world state spaces don't fit in memory. A 32×32 image alone is a 1,024-dim vector. Q-tables explode.
          

          

            Exploration vs exploitation
            Should the agent try new actions (explore) or repeat the known good ones (exploit)?
          

          

            Non-differentiability of the real world
            Real environments have noise, surprise, partial observability. Real reward functions may not even be differentiable.
          

          

            Agent modifies the environment while learning
            You can't collect a static dataset — every action changes future states. No batch mode for data collection.
          

        
      


      
Exploration vs exploitation strategies

      

        
StrategyBehavior

        
          

            ε-greedy (epsilon-greedy)
            With probability ε, take a random action (explore); with probability 1−ε, take argmax-Q (exploit). Decay ε over time.
          

          

            Softmax / Boltzmann
            Sample actions proportionally to their Q-values via a softmax — high-Q actions chosen more often, but never zero.
          

          

            Pure greedy
            Always argmax. Risk: agent locks onto a suboptimal action and never tries alternatives.
          

        
      


      

        
Restaurant analogy Prof

        
"Visit the same restaurant you like again and again, or sometimes try a different restaurant to maximize your reward function." That's exploration vs exploitation in everyday life.

      


      
To partially answer the high-state-count and continuous-domain challenges, classical RL is paired with neural networks — leading to Deep RL.

    

  


  

    

      
      
      
    


    

      
9. From Q-Tables to Deep RL (DQN)


      
Q-tables work fine for a 5×5 grid (25 states × 4 actions = 100 Q-entries). They do not scale to camera-driven robots or continuous control.


      

        
The clever trick Exam

        
Replace the Q-table with a neural network. The network takes the state (camera image, lidar, encoders) as input and outputs a Q-value for each action. This is Deep Q-Network (DQN).

      


      
Husky robot DQN example

      
    
        
  • Robot: Husky mobile robot with laser, camera, position encoders.
    • 
        
  • Method: DQN selects actions to find reward locations in a square arena.
    • 
        
  • Network: 3 hidden layers, trained with RMSProp, 400 iterations × 36 steps.
    • 
        
  • Output: predicted Q-values for each action primitive (forward, left, right, back).
    • 
        
  • Robot ends up moving like a rat — analogous to grid cells in the hippocampus from cognitive science.
    • 
      


      
Atari DQN — Mnih et al. 2015 Exam

      
The breakthrough paper — Human-level control through deep reinforcement learning, Nature 518:529–533.

      
    
        
  • Input: raw pixel state of an Atari game.
    • 
        
  • Output: action selection.
    • 
        
  • Result: matched or beat human performance on classic arcade games using only RL.
    • 
        
  • Sparked the modern Deep RL boom (AlphaGo, OpenAI Five, etc.).
    • 
      


      
Multi-agent emergent behavior — OpenAI Hide-and-Seek

      
The video shown in lecture: hiders & seekers trained via self-play. With only the simple rules "don't get spotted" / "spot them", agents spontaneously:

      
    
        
  1. Learn to chase / flee.
    2. 
        
  2. Hiders learn to build shelters from boxes.
    3. 
        
  3. Seekers learn to jump using ramps.
    4. 
        
  4. Hiders lock the ramps away.
    5. 
        
  5. Seekers discover "box surfing" by exploiting the physics engine.
    6. 
        
  6. Hiders respond by collecting all boxes before fortifying.
    7. 
      

      
None of this was incentivised — it emerged from competition + self-play co-evolution. Sometimes agents even find bugs in the physics engine and exploit them — a recurring sim-to-real headache.

Setup

  • Goal: reach the red charging station, no matter where the robot is initially placed.
    • 
        
  • Sensors (state inputs): color data, depth data, position encoders.
    •

OutcomeReward

Training loop

    2.

      •

12. Comparison Cheatsheet — LfD vs RL


      

        
DimensionLearning from DemonstrationReinforcement Learning

        
          

            Supervision signal
            Expert demonstrations (state→action pairs)
            Scalar reward
          

          

            Needs an expert?
            Yes — quality of demos = ceiling on quality of policy
            No — but reward function still hand-crafted
          

          

            Sample efficiency
            High (few demos can teach a task)
            Low (millions of trials typical)
          

          

            Generalisation beyond expert
            Hard — agent can't outperform demonstrator
            Possible — agent can discover novel strategies (AlphaGo move 37)
          

          

            Failure mode
            Garbage-in-garbage-out demos; covariate shift
            Reward hacking; sparse-reward stalls
          

          

            Best applied when…
            Task is hard to formalise but easy to demonstrate (cooking, folding clothes)
            Easy to evaluate, hard to solve (games, navigation, optimization)
          

        
      


      
On-policy vs off-policy Exam

      
Not heavily emphasised in lecture, but a likely fill-in-the-blank:

      

        
AlgorithmTypeUpdate target

        
          

            Q-learning
            Off-policy
            Uses max over next-state actions — assumes greedy future even if behavior is exploratory.
          

          

            SARSA
            On-policy
            Uses the actual next action the agent took (s, a, r, s', a') — learns the policy it is actually following.
          

        
      

      
Mnemonic: SARSA = the algorithm uses State-Action-Reward-State-Action — the next action it will actually take. Q-learning is bolder — it dreams of the optimal future even while exploring.
