1. Recap & Why a Brain at All?

      
Last week: sensors and actuators, plus Braitenberg vehicles — minimalist agents that already display "intelligent" behavior with almost no controller. This week we add the missing piece: the controller, which the prof uses interchangeably with the word "brain" to draw the analogy with biological systems.


      

        
Prof opener · Dennett quote Prof

        
"The juvenile sea squirt wanders through the sea searching for a suitable rock or hunk of coral to cling to and make its home for life. For this task, it has a rudimentary nervous system. When it finds its spot and takes root, it doesn't need its brain anymore, so it eats it!" — Daniel Dennett

        
Take-home: no movement → no need for a brain. Brains exist to generate action. The controller is the bridge between sensing and acting; without movement requirements, even biology drops it.

      


      
Control theory is a huge field — feedback control, adaptive control, robust control, model predictive control, swarm control, kinematics, RL. A full semester per topic would still not be enough. This lecture introduces the basics needed to wire a closed-loop controller for the project robot.


      

        
Prof emphasis Prof

        
"Today's lecture is one of the hardest topics in robotics. Make sure you understand the concepts and details — the slides can get dense." Translation: do not skim.

      

    

  


  

    

      
      
      
    


    

      
2. Situatedness & the Action–Perception Cycle


      

        
Definition Exam

        
Situatedness = the robot exists in a complex, dynamic, unstructured environment that strongly affects its behavior. Examples: a museum tour-guide robot navigating crowds, two robot teams playing soccer, a guide robot helping a vision-impaired child in a public space.

      


      
If the environment is complex + dynamic + unstructured, the robot is situated. If something changes, the robot must adapt — and that is exactly why we need a controller.


      
Exam question · situated vs non-situated Exam

      
The slide shows two pictures: (A) robots playing football against each other, (B) an industrial robot arm in a fixed cell. Which is situated, and why?

      
Answer: Figure A. The environment is dynamic (other agents moving), complex (ball trajectory, opponent strategy), and unstructured. The robot must react in real time to teammates, opponents, and ball state. Figure B has fixed obstacles and predetermined motions — non-situated.


      

        
Prof exam tip Prof

        
"In the exam, please read the full question. First state which figure is situated, then explain why. If you write the definition word-for-word, that's fine."

      


      
The action–perception loop

      
The endless loop every situated robot runs:

      
    
        
  1. Robot determines actuator state from current and previous sensor observations.
    2. 
        
  2. Action alters the robot/environment relation — and possibly the environment itself.
    3. 
        
  3. New sensor reading → next decision → next action. This generates behavior.
    4. 
      

      
Two roles of robot actions: (1) achieve goals, (2) alter the perceptual / physical environment. Both run continuously inside the loop.

    

  


  

    

      
      
    


    

      
3. What Is a Controller?


      
A controller is the algorithmic block between sensors and actuators. It uses perceived sensory information to decide which actions to perform.


      

        
BlockWhat it does

        
          
Perception (sensors)Reads the environment — encoders, camera, IR, touch, microphone.

          
Controller (brain)Maps sensor input → motor command using current + previous state.

          
Action (actuators)Motors / grippers execute the command and modify the environment.

        
      


      

        
Action vs Behavior Exam

        
Action = a motor command — go left, go right, raise leg, close gripper.

        
Behavior = what an external observer sees the robot doing — obstacle avoidance, wall following, light seeking. Behavior is composed of a sequence of actions.

        
The controller produces actions; the actions, taken together, produce behavior.

      


      
Why do we need a controller? Because the real world is dynamic, unpredictable, unstructured. To make a decision under those conditions, a situated robot needs an algorithm that can compute "what to do next" on the fly.

    

  


  

    

      
      
      
    


    

      
4. Open Loop vs Closed Loop


      

        
Core distinction Exam

        

          
Open loopClosed loop

          
            
ParametersSet in advance, fixedSensor data continuously influences behavior

            
Mechanism"Run for 30 s" — no feedbackUses error to generate action; goal is to minimise error

            
GuaranteesDesired result often not guaranteedAdapts to environment changes

            
ExampleMicrowave timerBraitenberg "lover" agent

            
Used in autonomous robots?RarelyYes — almost always

          
        

      


      
Open-loop · the microwave

      
You set the timer to 30 s. The microwave runs for 30 s regardless of whether the food is cold, frozen, or already burning. Semi-autonomous: needs user input, no awareness of the desired output. If the user gives the wrong time → burnt or cold meal.


      

        
Prof analogy · open-loop robot Prof

        
If you code a robot "move 40 cm forward, then rotate left", it works in a known empty space. Put an obstacle at 20 cm and the robot keeps trying to move 40 cm — motors grind, gears strip, battery drains. Open-loop fails the moment reality differs from the script.

      


      
Closed-loop · Braitenberg lover

      
The Braitenberg "lover" agent follows a moving light. When the light moves, the robot adapts. Why is this closed-loop?

      
    
        
  • It does not follow a pre-programmed plan.
    • 
        
  • It changes behavior based on real-time sensor input (light intensity).
    • 
        
  • If the light moves, the robot's trajectory changes — adaptive by construction.
    • 
      


      
Side question · is the Braitenberg lover autonomous? Exam

      
Yes. All three conditions from Week 1 are met: (1) no human is intervening with the robot directly — a human moves the light, not the robot, (2) the environment is unstructured, (3) it operates for a prolonged time. The human controlling the light is part of the environment, not part of the robot's command pipeline.

    

  


  

    

      
      
    


    

      
5. Kinematics — Forward vs Inverse


      
Two complementary tasks in robot kinematics. Both will be on the exam at the conceptual level; you will not have to solve any equations.


      

        
TypeGivenComputeDifficulty# solutions

        
          

            Forward kinematics
            Sequence of joint angles / motor commands
            Final position of the end-effector
            Easy (basic trigonometry)
            Always one
          

          

            Inverse kinematics
            Desired end-effector position in space
            Joint angle sequence to reach it
            Hard
            Zero, one, or many
          

        
      


      

        
Why inverse can have 0/1/many solutions Exam

        
    
          
  • Zero: the target is out of reach (too far, blocked).
    • 
          
  • One: rare — only one valid joint configuration exists.
    • 
          
  • Many: the same end-effector pose can be achieved by multiple joint configurations (think elbow-up vs elbow-down).
    • 
        

      


      

        
Prof scope note Prof

        
"This will be exam question. You will not solve anything, but you should be responsible for understanding the concept." Forward kinematics works out to e.g. x = L1·cos α + L2·cos(α+β); you don't need to memorise it, just know the structure.

      

    

  


  

    

      
      
    


    

      
6. Closed-Loop Feedback Control


      
The mechanical heart of every closed-loop controller. Three quantities — memorise these:


      

        
The three variables Exam

        

          
SymbolNameMeaning

          
            
rReference valueSpecification of the task / desired state. e.g. position relative to a shelf, distance of gripper to object.

            
yOutputThe actual measured state of the robot. e.g. distance to the object right now.

            
uControl valueWhat the controller outputs to the robot — e.g. motor power settings. Robots can't use r directly; r must be translated into u.

          
        

      


      

        
The error equation Exam

        
e = r − y

        
    
          
  • The output y is fed back to the controller.
    • 
          
  • y is compared with reference r → error e.
    • 
          
  • The control algorithm uses e to generate the control signal u, which drives the robot.
    • 
          
  • Goal: minimise e.
    • 
        

      


      
With each iteration, the error shrinks. The robot creeps toward the reference. e never quite hits zero in real hardware (more on that below) — but it gets within an acceptable threshold.


      

        
Prof emphasis Prof

        
"All focus is just how I can reduce my error to achieve my goal — that's the closed-loop feedback control mechanism for all robots." If you remember nothing else from this lecture, remember e = r − y.

      

    

  


  

    

      
      
      
    


    

      
7. On-Off (Bang-Bang) Control


      
The simplest closed-loop strategy. The robot is either at full speed or stopped — nothing in between.


      
    
        
  • Constant reference value r — e.g. "stop 10 cm from the object".
    • 
        
  • error = reference − measured.
    • 
        
  • Measured value comes from a proximity / distance sensor.
    • 
        
  • If error > 0 → motors forward. If error < 0 → motors reverse. If error = 0 → stop.
    • 
      


      
Why bang-bang almost never reaches zero error

      
In real hardware there is always sensor noise + software/hardware delay. Walk through the sequence:

      
    
        
  1. Robot approaches the object at full speed.
    2. 
        
  2. It crosses the reference distance — but reading the sensor and computing the error takes time.
    3. 
        
  3. By the time the controller decides "stop", the robot has overshot.
    4. 
        
  4. Now error is negative → motors reverse at full speed, overshoot back the other way.
    5. 
        
  5. Repeat forever. Plot looks like a triangle wave around r.
    6. 
      


      

        
Bang-bang downside Exam

        
On-off control generates high wear and tear on the motors (constant full-speed reversals) and cannot reach the exact reference. It oscillates around r forever.

      


      

        
Prof remark Prof

        
"In reality this is hard to achieve zero error. There will be some delay — I am reading my sensory information, processing it and making a decision — and that creates delay. I will never achieve zero error."

      

    

  


  

    

      
      
      
    


    

      
8. Proportional (P) Control


      
Fix the bang-bang oscillation by making the motor command proportional to the error: the closer you are, the slower you go.


      

        
P-controller equation Exam

        
u = K · e

        
K = the gain — the multiplicative factor of the control algorithm. Idea: "reduce your speed more as the error gets smaller".

      


      
Effect of gain on robot behaviour

      

        
GainBehaviorCurve shape

        
          

            Low gain
            Robot approaches reference slowly. May never quite reach r in available time.
            Dotted blue · slow rise
          

          

            High gain
            Robot approaches reference faster, reasonable settling.
            Dashed red · fast rise, slight overshoot
          

          

            Excessive gain
            Controller behaves like bang-bang again — oscillates around r.
            Green · oscillating, unstable
          

        
      


      

        
Key fact Exam

        
Oscillation = unstable controller. Tuning the gain is mostly trial-and-error: read papers using similar robots/tasks, try values, observe.

      


      
Beyond P · PI and PID

      

        
Prof scope note Prof

        
"We will not ask any questions in the final exam about the integral or derivative terms — only the proportional part." Slide 30 (PI / PID) is reading-only. If interested, see Matlab Control Systems videos and Peter Corke's book.

      


      
For completeness: PI adds an integral term that eliminates steady-state error (accumulates past error). PID adds a derivative term that damps overshoot (reacts to rate of change). Industrial sweet spot — and a lot of money in the industrial domain depends on tuning these well.

    

  


  

    

      
      
      
    


    

      
9. Robotic Paradigms — Sense, Plan, Act


      
Beyond the math of feedback, we need a mental model of how a situated robot operates. Welcome to robotic paradigms.


      
What is a paradigm?

      
"A paradigm is a basic framework of assumptions, principles and methods from which the members of the community work." Different camps in robotics solve the same problem with very different paradigms (hand-coded, learning-based, RL, classical control). All can be correct — but you should pick one consciously for your project.


      
Closed world vs open world Exam

      

        
AssumptionWhat it meansFailure mode

        
          

            Closed world
            The robot has access to a world model containing everything it needs to know to accomplish the task.
            "Success of the robot depends on how well the human programmer can think of everything."
          

          

            Open world
            World model is dynamic; robot cannot access everything it needs.
            "Closed-world assumption cannot be applied to that particular domain."
          

        
      


      

        
Exam pattern · which is open world? Exam

        
The slide compares two robot teams. Robot soccer = open world (multiple agents, unpredictable opponents, dynamic ball). Synchronised robot dance = closed world (prescribed maneuvers, fixed sensors, no surprises). Justification: open-world environments are too complex to be modelled by a single human programmer.

      


      
Robot primitives

      
Like primitive data types in programming (int, float, byte), robotics has three primitives — every paradigm is built by combining them:

      
    
        
  1. Sense — gather information using sensors (camera matrices, joint encoders, IR distance, etc.).
    2. 
        
  2. Plan — create a world model and decide the next move (logic / neural / rule-based engine).
    3. 
        
  3. Act — generate actuator commands to execute the action.
    4. 
      

      
The paradigm = how you wire these three primitives together. Not every paradigm uses all three.

    

  


  

    

      
      
    


    

      
10. Hierarchical (Deliberative) Paradigm


      
The oldest paradigm. Wiring: Sense → Plan → Act, strictly sequential. Every cycle: observe everything, build a full world model, plan the next move, then execute it.


      
    
        
  • Sense — provides full representation of the environment.
    • 
        
  • Plan — logic / neural / rule-based engine decides actions from that representation.
    • 
        
  • Act — the robot's motor behaviour executes the plan.
    • 
      


      
Strengths & weaknesses

      

        
+ Strengths− Weaknesses

        
          

            Accurate once the plan is computed
            Slow — one full sense+plan+act cycle per action
          

          

            Good for known, static environments
            Ignores the dynamic aspect of the environment
          

          

            Explicit world model — debuggable
            Not biologically realistic — no reflex-like fast loops
          

        
      


      

        
Historical anchor · Shakey Prof

        
Shakey the Robot (SRI, late 1960s) was the canonical hierarchical robot. The GoTo program planned a route by computing whether obstacles lay on a direct path. If they did, sub-goals near the obstacle were inserted. Once the plan was complete, Shakey executed it.

        
Quote from Introduction to AI Robotics: "Unfortunately, the flow of events ignored biological evidence that sensed information can be directly coupled to an action, which is why the sensed information input is blacked out."

      


      
If your robot is a Shakey-style system with many sensors, every action requires re-sensing-replanning-reacting. As the sensor count and computation grow, the system becomes physically infeasible or extremely slow.

    

  


  

    

      
      
    


    

      
11. Reactive Paradigm


      
Rodney Brooks' provocation: remove planning. Wiring shrinks to Sense → Act. Sensor signals are filtered/modulated and routed directly to actuators.


      
    
        
  • Connects action with sensing — no internal world model.
    • 
        
  • Sensors produce continuous signals; after light modulation (e.g. a filter), they enable the robot to act.
    • 
        
  • Biological agents display intelligent behaviour with this paradigm — e.g. avoiding light, tracking prey.
    • 
        
  • Braitenberg vehicles are reactive controllers by construction.
    • 
      


      
Trade-offs

      

        
+ Strengths− Weaknesses

        
          

            Very fast — short tight loop, no deliberation
            Cannot save information — no internal state
          

          

            Biologically plausible
            No internal representation of the environment
          

          

            No advanced algorithm knowledge required
            Cannot learn from experience
          

        
      


      

        
Prof example · the fly that won't quit Prof

        
"For a biological agent with reactive control, it doesn't matter if it's a virtual fly or something is moving in front of it — it will just attack it. There is no learning. Something is changing, but you are behaving the same way." Useful behaviour, but no adaptation.

      

    

  


  

    

      
    


    

      
12. Hybrid Deliberative / Reactive Paradigm


      
If hierarchical is too slow and reactive can't learn — combine them.


      

        
Hybrid wiring Exam

        
[First] Hierarchical/Deliberative → Plan

        
[Then] Reactive → Sense-Act

        
You plan first (using deliberative reasoning, world model), then run a reactive sense-act loop while executing — fast obstacle response built on top of a coherent global plan.

      


      
Why combine?

      
    
        
  • Get speed from the reactive sense-act loop (handles dynamic environments in real time).
    • 
        
  • Get adaptability + planning from the deliberative layer (long-horizon goals, world model).
    • 
        
  • Caveat: requires parallel processing — plan and react must run side-by-side. More engineering effort.
    • 
      


      
Side-by-side Exam

      

        
ParadigmWiringSpeedMemory / learningReal-world fit

        
          

            Hierarchical
            Sense → Plan → Act
            Slow
            Yes (world model)
            Static, known
          

          

            Reactive
            Sense → Act
            Very fast
            None
            Dynamic but simple
          

          

            Hybrid
            Plan, then Sense-Act in parallel
            Fast (reactive layer) + planned
            Yes
            Best balance
          

        
      


      

        
Prof framing Prof

        
"We are orthogonally planning the task — getting benefits of being fast in the environment plus learning the environment dynamics." For your project robot, hybrid is usually the right target.

      

    

  


  

    

      
    


    

      
13. Worked Example — "Make Robot #7 Kick the Ball"


      
Slide 36 ends with this exam-style prompt: As a roboticist, your assignment is to enable robot #7 to kick a ball. Detail the specific robot primitives — Sensing, Planning, and Acting — that robot #7 will use, including examples for each stage.


      

        
Sample answer template Exam

        

          
PrimitiveFor "kick the ball"

          
            

              Sense
              Camera to detect the ball position and the goal; joint encoders to know each leg joint angle; possibly an IMU for balance.
            

            

              Plan
              If ball is in front of the robot at kicking distance → raise the kicking leg, swing it through the ball trajectory. Could be hand-coded rules, a state machine, or an RL policy. (Note: "in the real robot reinforcement learning will almost always fail" — prof.)
            

            

              Act
              Joint-position commands to leg motors that raise the leg and swing it through; balance corrections via the support leg.
            

          
        

      


      
Bonus concept the prof name-dropped during a student question: motion primitives — when a high-level controller ("press the buy button" / "kick the ball") delegates to a lower-level controller that handles the fine motor sequence. Cognition picks the goal; motion primitives execute the trajectory.

    

  


  

    

      
    


    

      
14. Practical Tie-In · Arduino as the Brain


      
The lecture closes by physically pointing at the controller in your kit: the Arduino board is the brain of your robot. The TinkerCad demos showed:


      
    
        
  • A bare circuit (LED + button) has no controller — a human is the brain pressing the button.
    • 
        
  • Add an Arduino + photoresistor → the Arduino reads light intensity and modulates the LED. That is a closed-loop reactive controller.
    • 
        
  • Replace the LED with a motor + add two photoresistors → you have a Braitenberg vehicle on an Arduino.
    • 
        
  • You should drive motors through a motor controller / H-bridge, not directly off Arduino pins — higher voltage with lower-voltage logic. Otherwise the board burns.

Key Takeaways

    
    
      
  1. Brain = controller. No movement, no need for a brain (Dennett's sea squirt). The controller maps sensor input to actuator output and is what makes situated behaviour possible.
    2. 
      
  2. Action ≠ Behavior. Action = motor command (go left). Behavior = what observers see (obstacle avoidance). Behavior is composed of actions. Likely fill-in-the-blank exam target.
    3. 
      
  3. Open loop = parameters fixed in advance, no feedback (microwave). Closed loop = sensor data continuously influences behavior, error drives control (Braitenberg lover). Autonomous robots use closed loop.
    4. 
      
  4. Memorise e = r − y. r = reference (desired), y = output (measured), u = control signal. Goal: minimise e. This equation is the heart of feedback control.
    5. 
      
  5. Bang-bang oscillates. Hardware delay + sensor noise → robot overshoots r forever. Generates motor wear. Never reaches zero error.
    6. 
      
  6. Proportional control · u = K · e. Slow down as you approach. Low gain = sluggish; high gain = fast; excessive gain → bang-bang again (oscillation = unstable). Only the P term is on the exam.
    7. 
      
  7. Forward kinematics easy (joint angles → end position, basic trig). Inverse kinematics hard (end position → joint angles; can have 0, 1, or many solutions). You will not solve, but must understand the concept.
    8. 
      
  8. Three robot primitives: Sense, Plan, Act. The paradigm is how you wire them.
    9. 
      
  9. Hierarchical (Sense→Plan→Act): accurate, slow, ignores dynamics — Shakey. Reactive (Sense→Act): fast, biological, no memory or learning — Braitenberg. Hybrid: plan first, then run reactive loop in parallel — best for situated robots.
    10. 
      
  10. Closed-world assumption = robot has full world model (programmer thought of everything). Open-world = dynamic environment, robot can't access everything. Open-world ⇒ situated robot ⇒ closed-loop control + reactive/hybrid paradigm.
